# CODING_STANDARDS.md

# URL Shortener SaaS Platform
## Frontend Coding Standards

Version: 1.0

---

# Purpose

This document defines the engineering standards for the frontend.

Every feature generated by AI or implemented manually must follow these
rules.

These standards exist to ensure:

- Readability
- Maintainability
- Reusability
- Consistency
- Scalability

When in doubt,

follow these standards instead of generating "creative" solutions.

---

# General Philosophy

Write code for humans first.

Optimize for maintainability rather than cleverness.

Every file should be easy to understand by another developer.

---

# Engineering Principles

Always prioritize

Simplicity

↓

Readability

↓

Reusability

↓

Performance

↓

Optimization

Never sacrifice readability for minor optimizations.

---

# React Standards

Use

✓ Functional Components

✓ React Hooks

✓ ES Modules

✓ Modern JavaScript

Never use

✗ Class Components

✗ Legacy Lifecycle Methods

✗ Deprecated APIs

# Folder Rules

Never place unrelated files together.

Components belong in

components/

Pages belong in

pages/

Services belong in

services/

Utilities belong in

utils/

---

# Imports

Order imports

React

↓

Libraries

↓

Components

↓

Hooks

↓

Services

↓

Utilities

↓

Styles

Never use circular imports.

Prefer absolute imports where configured.

---

# Styling

Use

Tailwind CSS

Never use

Inline CSS

Style attributes

Hardcoded colors

Hardcoded spacing

Hardcoded font sizes

Spacing must follow DESIGN.md.

---

# CSS Rules

Prefer

Tailwind utilities

Create reusable utility classes only when necessary.

Never duplicate utility chains repeatedly.

---

# Icons

Use

Lucide React

Only.

Avoid mixing icon libraries.

---

# State Management

Global UI State

↓

Zustand

Server State

↓

React Query

Component State

↓

useState

Forms

↓

React Hook Form

Never store server data inside Zustand.

# API Rules

Pages never communicate directly with Axios.

Correct

Page

↓

Service

↓

Axios Instance

↓

Backend

Incorrect

Page

↓

axios.get()

↓

Backend

---

# Axios Standards

All API calls use

axiosInstance

Responsibilities

JWT

Interceptors

Timeouts

Base URL

Error Handling

Authorization

Never create multiple Axios instances.

---

# Error Handling

Every API call should handle

Loading

Success

Failure

Empty State

Network Failure

Unauthorized

Use Toast notifications.

Never swallow errors silently.

---

# Forms

Every form should

Validate inputs

Display field errors

Prevent duplicate submissions

Disable button while submitting

Show success feedback

Never rely only on backend validation.

---

# Tables

Tables must support

Loading

Empty State

Pagination

Sorting

Search

Responsive Layout

Hover States

Selection (future)

# Loading Strategy

Prefer

Skeletons

Instead of

Lazy Loading

Memoization

Pagination

Debouncing

Code Splitting

# Security

Expose secrets

Hardcode API URLs

# AI Restrictions

Never

Generate placeholder components that are never used.

Generate duplicate layouts.

Hardcode colors.

Hardcode spacing.

Hardcode API endpoints.

Duplicate authentication logic.

Duplicate Axios configuration.

Introduce unnecessary dependencies.

Modify backend code unless explicitly requested.

---

# Code Review Checklist

Every feature should answer

Is the code reusable?

Is the component small?

Is the logic separated?

Does it follow DESIGN.md?

Does it follow ARCHITECTURE.md?

Would another developer understand this in six months?

If any answer is "No",

refactor before committing.

---

# Future Maintainability

Every new feature should integrate naturally into the existing codebase.

The architecture should improve over time,

never become more complicated.

Consistency is always preferred over cleverness.

---

# Final Principle

Write code that feels like it belongs to the project.

Every generated file should look as if it was written by the same engineer.

The codebase should grow through reusable patterns,

not duplicated implementations.
